    üöÄ Performance Notes (GoMT4)¶

    Practical tips to keep GoMT4 fast, stable, and resource‚Äëefficient. Focus is on your current codebase (examples/, pb API, MT4 terminal on Windows).

    üéØ Goals¶

    • Low latency for quotes/updates.
    • Throughput for bulk reads (history, many symbols).
    • Stability under loss/reconnects.
    • Predictable CPU/RAM usage.

    üî• Hot paths to watch¶

    • Streaming: StreamQuotes, StreamOpenedOrderProfits, StreamTradeUpdates.
    • Batch RPCs: ShowQuotesMany, paged/streamed Orders History (ShowOrdersHistory, StreamOrdersHistoryExample).
    • Symbol metadata: ShowSymbolParams, ShowTickValues used in loops.

    Keep these tight: avoid heavy logging, allocations, and repeated RPCs.

    üì¶ Batch, don‚Äôt loop RPCs¶

    • Prefer batch APIs over per‚Äësymbol calls:

    • ‚úÖ ShowQuotesMany instead of ShowQuote in a loop.

    • History: use paged/streaming endpoints rather than huge single responses.

    • ‚úÖ OrdersHistory or StreamOrdersHistoryExample.

    Why: fewer round‚Äëtrips ‚Üí lower latency and CPU per message.

    üì° Streaming tuning¶

    • Backpressure: use the server‚Äôs knobs when available.

    • Example: OpenedOrdersProfitStreamRequest{ buffer_size } ‚Äî set buffer high enough to absorb short spikes, but not so high that consumer lags unnoticed.

    • Consumer loop: drain fast, do work in a goroutine pool if processing is heavy.
    • Tick throttling: if you don‚Äôt need every tick, aggregate with a time.Ticker (e.g., 100‚Äì200ms) before updating UI/logs.
    • Timeouts: long‚Äëlived streams should use WithCancel, not short timeouts; add an idle watchdog to reconnect when no data for X seconds.
    ctx, cancel := context.WithCancel(parent)
defer cancel()
updates, errCh := svc.StreamQuotes(ctx, symbols)
for {
    select {
    case u := <-updates: /* cheap handling */
    case err := <-errCh: /* reconnect + backoff */
    case <-idle.C:      /* no data ‚Üí ping/refresh */
    }
}

    üßµ Concurrency patterns (Go)¶

    • One gRPC connection per MT4 instance; reuse *grpc.ClientConn (creating many is expensive).
    • Worker pool for CPU‚Äëbound post‚Äëprocessing (e.g., statistics on history), not for issuing parallel trade RPCs to the same MT4 terminal.
    • Channel fan‚Äëout: if several consumers need the same stream, broadcast from one reader goroutine; don‚Äôt open multiple identical streams.

    üß∞ Symbol metadata cache¶

    • Cache ShowSymbolParams and ShowTickValues in memory (map by symbol). These values change rarely; re‚Äëfetch only on symbol list changes or once per session.
    • Use cached digits/point/lot_step for rounding before any trade call (see Cookbook ‚Üí RoundVolumePrice.md).

    üßÆ History I/O¶

    • Request time windows in chunks (days/weeks) and merge client‚Äëside.
    • Prefer H1/H4/D1 where possible; M1 over long ranges is expensive.
    • For analytics, stream and process as you go instead of storing everything first.

    üóúÔ∏è Compression & payloads¶

    • gRPC compression helps on large history responses; avoid compressing ultra‚Äëshort ticks.
    • Avoid logging full payloads in hot paths; log only counts and latency metrics.

    üß† Allocations & GC¶

    • Reuse buffers/slices for quote aggregation; pre‚Äëallocate with make(..., cap) when size is known.
    • Convert time ‚Üî Timestamp carefully in tight loops; avoid repeated parsing/formatting.
    • Use strings.Builder for building CSV/TSV outputs.

    ‚è±Ô∏è Timeouts & retries¶

    • Unary calls: context.WithTimeout (100ms‚Äì3s typical for local terminal). Add idempotency where safe and exponential backoff.
    • Streams: no short deadline; reconnect on error/idle with jittered backoff (see Cookbook ‚Üí HandleReconnect.md, UnaryRetries.md).

    üñ•Ô∏è Windows & MT4 specifics¶

    • Keep MT4 responsive: exclude MT4 data directory from real‚Äëtime antivirus scanning if safe in your environment.
    • Power plan: use High performance to avoid CPU sleep throttling for low‚Äëlatency streams.
    • Symbol visibility: ensure symbols are visible before quote/history calls (Cookbook ‚Üí EnsureSymbolVisible.md). Pre‚Äësubscribe to reduce first‚Äëtick latency.

    üìä Observability¶

    • Add lightweight metrics: per‚ÄëRPC latency, stream message rate, reconnect counters.
    • Sampled logs (1/N) for hot paths; full debug logs only on demand.

    ‚úÖ Checklist (TL;DR)¶

    • [ ] Use batch requests and paged/streamed history.
    • [ ] Keep one gRPC conn; reuse clients.
    • [ ] Cache SymbolParams / TickValues.
    • [ ] Fast stream consumers; apply throttling/aggregation if UI‚Äëfacing.
    • [ ] Timeouts for unary; cancel for streams; exponential backoff with jitter.
    • [ ] Minimal logging in hot paths; metrics for rates/latency.
    • [ ] Pre‚Äëmake symbols visible or subscribe early.
    • [ ] Prefer higher‚ÄëTF history when possible; chunk long ranges.
